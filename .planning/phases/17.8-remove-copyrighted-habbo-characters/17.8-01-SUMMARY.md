## Decisions Made

- PixelLab MCP tools were unavailable, so team characters were created by hue-rotating the existing beanie-hoodie-guy PNG. Each team gets a distinct primary color: planning=navy(220), infrastructure=orange(30), support=teal(165). Core-dev is a direct copy (unchanged hoodie look).
- `getAtlasForTeam()` exported as a named function for testability and potential use in the character legend panel (Plan 03).
- `createRenderable()` has a double-fallback: tries team-specific atlas first, then falls back to default `pixellab` atlas, ensuring no avatars disappear if a team atlas fails to load.
- `(spec as any).team` used to read team without modifying AvatarSpec type — will be cleaned up in Plan 02.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Added fallback to default atlas in createRenderable()**
- **Found during:** Task 2 (wiring renderer)
- **Issue:** Plan only specified team atlas fallback to `pixellab` in the `isAvailable()` check. The `createRenderable()` call would silently return null if a team atlas failed to load, even though the default atlas is available.
- **Fix:** Added secondary fallback in `createRenderable()` — tries team atlas first, then falls back to `DEFAULT_ATLAS_NAME` for both the target frame key and the rotation fallback frame.
- **Files modified:** src/pixelLabAvatarRenderer.ts
- **Verification:** TypeScript compiles clean (0 errors in src/)
- **Committed in:** 0103f2c (Task 2 commit)
