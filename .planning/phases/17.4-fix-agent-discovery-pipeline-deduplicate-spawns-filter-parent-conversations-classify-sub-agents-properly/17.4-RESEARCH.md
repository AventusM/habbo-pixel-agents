# Phase 17.4: Fix Agent Discovery Pipeline - Research

**Researched:** 2026-03-09
**Domain:** Agent discovery, deduplication, JSONL filtering, sub-agent classification
**Confidence:** HIGH

## Summary

The agent discovery pipeline in `src/agentManager.ts` has three distinct bugs that cause incorrect avatar spawning in the room canvas:

1. **Duplicate spawns**: When `discoverAgents()` runs during the `ready` event, it fires `agentCreated` for each discovered JSONL file. If the webview later sends `requestAgents`, the extension re-broadcasts `agentCreated` for all agents. While `avatarManager.spawnAvatar` guards against duplicate avatar creation (returns existing avatar), the `RoomCanvas.tsx` `agentCreated` handler still runs side effects (orchestration add, section assign, idle wander start) on every message.

2. **Parent conversation JSONL tracked as agents**: The Claude Code project directory contains both `UUID.jsonl` files (top-level/parent conversations) and `UUID/subagents/*.jsonl` files (actual sub-agents). Currently `discoverAgents()` tracks ALL `.jsonl` files in the root directory, including parent conversations. Parent conversations are the orchestrator -- they should not appear as room avatars. Only files inside `subagents/` directories are actual working agents.

3. **Sub-agent classification ignores meta.json**: Each sub-agent JSONL has a companion `*.meta.json` file (e.g., `agent-abc123.meta.json`) containing `{"agentType":"gsd-executor"}`. The current code reads the first 50 lines of the JSONL to extract `subagent_type` from `Agent` tool_use blocks, which is unreliable because: (a) the JSONL is the sub-agent's own transcript, not the parent's, so it won't contain the `Agent` tool_use that spawned it; (b) the meta.json file is the authoritative source provided by Claude Code itself.

**Primary recommendation:** Fix all three bugs in `agentManager.ts` -- skip root-level JSONL files that have a matching session directory (they are parent conversations), read `meta.json` for sub-agent classification, and add deduplication guards in the `agentCreated` handler in `RoomCanvas.tsx`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| JSONL subagent type detection | Custom transcript scanning | meta.json `agentType` field | Authoritative source; transcript scanning is fragile |
| Parent vs sub-agent detection | Content-based heuristics | Directory structure check (has sibling dir?) | Structural signal is 100% reliable |

## Common Pitfalls

### Pitfall 1: Standalone Sessions Without Subagents
**What goes wrong:** Filtering ALL root-level JSONL files would hide standalone sessions (no subagents directory) that ARE legitimate single-agent sessions.
**Why it happens:** Not all sessions spawn sub-agents. Some are direct user conversations.
**How to avoid:** Only filter root JSONL files that have a matching session directory. Standalone `.jsonl` files (no sibling directory) could be legitimate agents IF they are recently active. However, consider whether standalone sessions should show as avatars at all -- they might be the user's own conversation.
**Recommendation:** For this phase, filter out ALL root-level JSONL files. Only track files inside `subagents/` directories. Root-level JSONLs are always the parent/orchestrator conversation.

### Pitfall 2: Race Condition Between discoverAgents and fs.watch
**What goes wrong:** `fs.watch` callback fires for a file that `discoverAgents()` already tracked.
**Why it happens:** The `trackAgent` guard (`if (this.agents.has(jsonlPath)) return`) already handles this.
**How to avoid:** Keep the existing guard. No additional work needed here.

### Pitfall 3: meta.json May Not Exist Yet When JSONL Appears
**What goes wrong:** The watcher detects a new JSONL file but the meta.json hasn't been written yet.
**Why it happens:** Claude Code may write the JSONL first and meta.json slightly after.
**How to avoid:** Fall back to JSONL content scanning when meta.json is missing. Consider a short retry or watching for meta.json appearance.

### Pitfall 4: agentCreated Side Effects Run on Re-broadcast
**What goes wrong:** `orchestrationAddAgent`, `sectionManager.assignAgent`, `idleWander.setAgentRole` all run again on duplicate `agentCreated`.
**Why it happens:** RoomCanvas handler doesn't check if avatar already exists before running side effects.
**How to avoid:** Add early return when `avatarManager.getAvatar(msg.agentId)` returns truthy.
