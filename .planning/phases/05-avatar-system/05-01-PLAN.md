Create `src/isoAvatarRenderer.ts` implementing avatar sprite rendering with 8-direction support and 3-4 layer composition system.

**Required exports:**

1. **AvatarSpec interface** - Avatar specification with variant system:
```typescript
export interface AvatarSpec {
  /** Unique avatar ID */
  id: string;
  /** Tile X coordinate */
  tileX: number;
  /** Tile Y coordinate */
  tileY: number;
  /** Tile Z coordinate (height level 0-9) */
  tileZ: number;
  /** Habbo direction (0-7 clockwise from NE) */
  direction: 0 | 1 | 2 | 3 | 4 | 5 | 6 | 7;
  /** Palette variant (0-5 for 6 distinct characters) */
  variant: 0 | 1 | 2 | 3 | 4 | 5;
  /** Animation state ('idle' | 'walk') */
  state: 'idle' | 'walk';
  /** Animation frame (0-3 for walk cycle, 0 for idle) */
  frame: number;
}
```

2. **createAvatarRenderable function** - Returns Renderable with multi-layer composition:
```typescript
export function createAvatarRenderable(
  spec: AvatarSpec,
  spriteCache: SpriteCache,
  atlasName: string
): Renderable
```

**Implementation details:**

1. **Frame key format**: `avatar_{variant}_{layer}_{direction}_{state}_{frame}`
   - Example: `avatar_0_body_2_idle_0`, `avatar_3_hair_5_walk_2`

2. **Layer rendering order** (back to front):
   - Layer 1: `body` (skin tone)
   - Layer 2: `clothing` (shirt/outfit)
   - Layer 3: `head` (face)
   - Layer 4: `hair` (hairstyle)

3. **8-direction support**: All directions 0-7 must be distinct sprites (no mirroring like furniture). Habbo avatars have unique sprites for each direction.

4. **Variant system**: Use `spec.variant` (0-5) to select different color palettes. Each variant should load different sprite sets representing different skin tones or clothing colors.

5. **Sprite positioning**: Same as furniture renderer pattern:
   - Convert tile to screen: `tileToScreen(spec.tileX, spec.tileY, spec.tileZ)`
   - Center horizontally: `dx = Math.floor(screen.x - frame.w / 2)`
   - Align bottom to tile: `dy = Math.floor(screen.y - frame.h)`

6. **Graceful degradation**: If any layer frame is missing, log warning and skip that layer (don't crash the render loop).

7. **Debug logging**: Log once per avatar ID when first rendered (same pattern as furniture renderer).

**Create smoke tests in `tests/isoAvatarRenderer.test.ts`:**

1. Test AvatarSpec interface exists with all required fields
2. Test createAvatarRenderable returns Renderable with correct tileX/tileY/tileZ
3. Test renderable.draw() doesn't throw when sprite cache is empty (graceful degradation)
4. Test all 8 directions produce distinct frame lookups (0-7)
5. Test all 6 variants produce distinct frame lookups (0-5)
6. Test idle vs walk state produces different frame keys

Use happy-dom environment and OffscreenCanvas mock from Phase 2 pattern. Minimum 6 test cases.

**DO NOT**: Implement actual sprite sourcing/creation yet - that's Task 2. Focus on the rendering logic with assumption sprites will exist.

**WHY NOT mirrors**: Furniture uses 4 directions with mirrors because office furniture is symmetrical. Avatars need 8 unique directions because characters are asymmetric (different arm/leg positions when walking diagonally).

Create placeholder avatar sprites following the same approach as Phase 4 furniture placeholders, but adapted for 8 directions × 6 variants × 4 layers × 2 states.

**Sprite requirements:**

1. **Size**: 96×128px per sprite (taller than furniture to represent human figure)
2. **Atlas layout**: Organize as grid to fit all combinations
3. **Variants**: 6 distinct color schemes:
   - Variant 0: RED body, BLUE hair
   - Variant 1: CYAN body, ORANGE hair
   - Variant 2: GREEN body, YELLOW hair
   - Variant 3: MAGENTA body, WHITE hair
   - Variant 4: ORANGE body, CYAN hair
   - Variant 5: BLUE body, GREEN hair

4. **Layers** (simplified for placeholders):
   - Body layer: Solid color oval (represents torso)
   - Clothing layer: 20% darker shade of body color (represents shirt)
   - Head layer: Lighter circle on top (represents face)
   - Hair layer: Distinct color rectangle on top (represents hairstyle)

5. **Direction indicators**: Draw directional arrow or number on each sprite to show which direction it represents (0-7).

6. **State handling** (simplified for Plan 01):
   - Only generate `idle` state with frame 0
   - Walk cycle frames will be added in Plan 02
   - Frame key: `avatar_{variant}_{layer}_{direction}_idle_0`

**Script: `scripts/generate-avatar-placeholders.sh`**

Use ImageMagick to generate sprites. Example approach:

```bash
#!/bin/bash
# Generate 6 variants × 8 directions × 4 layers × 1 state = 192 sprites

variants=(
  "red:blue"      # 0: red body, blue hair
  "cyan:orange"   # 1: cyan body, orange hair
  # ... etc
)

for v in {0..5}; do
  for dir in {0..7}; do
    # Generate body layer
    convert -size 96x128 xc:transparent \
      -fill "${body_color}" -draw "ellipse 48,80 30,40 0,360" \
      -pointsize 24 -fill white -annotate +35+70 "$dir" \
      "avatar_${v}_body_${dir}_idle_0.png"

    # Generate clothing, head, hair layers similarly
    # ...
  done
done

# Pack into atlas using Texture Packer CLI or montage
montage avatar_*.png -geometry 96x128+0+0 -tile 16x12 -background transparent avatar_atlas.png

# Generate JSON manifest
# ...
```

**Manifest format**: Texture Packer JSON hash format matching furniture atlas pattern:

```json
{
  "frames": {
    "avatar_0_body_0_idle_0": { "frame": { "x": 0, "y": 0, "w": 96, "h": 128 }, ... },
    "avatar_0_clothing_0_idle_0": { "frame": { "x": 96, "y": 0, "w": 96, "h": 128 }, ... },
    // ... all 192 frames
  },
  "meta": {
    "image": "avatar_atlas.png",
    "size": { "w": 1536, "h": 1536 }
  }
}
```

**Atlas dimensions**: 16 sprites wide × 12 sprites tall = 192 sprites total (1536×1536px atlas).
